# Aury v0.2 roadmap

Tracking doc for the v0.2 language milestone. v0.1 proved the language can
express real programs and that every gate *emits* a repair. v0.2 closes the
loop it left open: the Track 4 evaluation showed that the only case where the
repair loop *mechanically converges to the correct program* is a **parse
deficit** — effect-leak repair is not mechanically applied, and the region /
affine machinery is still a pass-through no-op. So v0.2 has one theme:

> **Make the repair loop mechanically converge on the *structural* gates it
> currently only diagnoses (effects, regions), give the language enough range
> to be worth measuring, then measure it against a real baseline.**

Ordered by dependency and leverage. The invariant for every track is unchanged:
**the interpreter is the semantics, and the native/wasm backend must produce the
identical observable value** (differential parity is a test failure, not a
warning). Every new gate rejection must ship with an *admissible mechanical
repair* and a test proving the loop applies it.

Legend: `[ ]` todo · `[~]` in progress · `[x]` done

---

## Track A — Effect/capability repair (anchor; smallest diff, most on-thesis) — ✅ CORE DONE

Repair-convergence goal met: all three effect-gate directions (widen / drop-unused
/ unknown-cap) are mechanically applied by the loop, and the eval corpus has a
genuine effect-gate convergence case. Deferred to a later slice (user decision
2026-07-17): the `with` capability-scope form + `wrap-in-capability-scope` repair
(needs capability *tokens* — v0 `rng` is ambient, so a sound `with` is its own
design), and effectful ops for the non-`rng` caps (`clock.now`, `log.i64`) with
native/wasm parity.

The most thesis-critical hole. v0.1 has effect rows and deterministic `rng`
gating, but an effect mismatch is only *diagnosed* — the loop cannot mechanically
close it, because there is no signature-widening / capability-scope repair
template. This track makes effect rejections converge.

- [~] Promote effect rows to first-class capabilities beyond `rng`: `(fs read)` /
      `(fs write)`, `(net)`, `(clock)`, `(state)`, `(log)` as capability atoms in
      the effect row (`src/types.rs`, `src/ast.rs`, `src/json.rs`). Interpreter
      keeps them deterministic / stubbed where no real OS shim exists.
      - [x] Fixed capability **vocabulary** (`KNOWN_CAPABILITIES` in validate.rs):
            `rng, clock, log, net, state, fs read, fs write`. An out-of-vocabulary
            cap is an `UNKNOWN_CAPABILITY` rejection with a mechanical drop repair.
      - [ ] Effectful **ops** for the new caps (e.g. `clock.now`, `log.i64`) with
            interp + native + wasm parity — not yet; only `rng.*` produces effects.
- [x] Effect inference-free checking, all three directions carrying mechanical
      repairs the loop applies (`src/validate.rs`): under-declared →
      `widen_effect_row`; over-declared (least-privilege) → `EFFECT_OVER_DECLARED`
      / `drop_unused_effect` (narrowing to pure removes the clause); unknown cap →
      `UNKNOWN_CAPABILITY` / drop.
- [~] **Mechanical repair templates** for the effect gate (`src/repair.rs`,
      `src/loop_driver.rs`): `widen-effect-row` (add the missing capability to the
      caller's `(effects …)`), `wrap-in-capability-scope` (introduce a `with`
      form), and `drop-unused-effect`. The loop must *apply* the top-ranked repair
      and re-enter, not just print it.
      - [x] `widen_effect_row` **DONE**: the `EFFECT_EXCEEDS_DECLARED` /
            `CALL_EFFECT_EXCEEDS_DECLARED` rejections now target the `(fn …)` node
            and carry a `with` = the corrected `(effects …)` clause; the driver's
            `apply_repair` special-cases it via `set_effect_row_by_id` (replaces
            an existing clause or inserts one before requires/ensures/body). Closes
            the v0.1 "effect-leak repair is NOT mechanically applied" gap. New
            tests `repair_loop_mechanically_widens_effect_row` +
            `repair_loop_widens_existing_effect_row_in_place`; suite green at 49.
      - [x] `drop-unused-effect` **DONE** (over-declaration + unknown-cap both use
            it; see the effect-checking item above).
      - [~] `wrap-in-capability-scope` — DEFERRED with the `with` form (user
            decision 2026-07-17); needs capability tokens.
- [ ] `with` form: `(with (cap …) body)` grants a capability to a lexical scope;
      parse + type + interp + lower (`src/ast.rs`, `src/validate.rs`,
      `src/interp.rs`, `src/lower.rs`). This is the missing template the Track 4
      note called out.
- [ ] Tests: an under-declared function is rejected with `widen-effect-row`, the
      loop applies it, and the corrected program is accepted end to end
      (`tests/integration.rs`). Differential parity for any capability with an
      observable value.
- [x] **Acceptance (partially met):** a program that uses an effectful op without
      declaring the effect is *mechanically repaired by the loop* (not hand-fixed),
      and the eval corpus gained a genuine effect-gate convergence case
      (`examples/agent/effect-leak.aury` → corpus `effect-leak`: first-shot `type✗`
      → accepted in 1 patch, oracle 2/2; rescued-by-repair count in the corpus is
      now 2, not 1). Remaining for full acceptance: capability vocabulary +
      `with` scope form below.
- **Out of scope:** real OS execution / audited extern shims (still deferred);
      capabilities are checked and deterministically modeled, not sandboxed.

## Track B — Growable aggregates + affine move-tracking (`vec push`) — 🚧 IN PROGRESS

The affine/region design is currently a pass-through no-op because nothing
mutates an aggregate. `push` over a loop-carried vec finally exercises
move-tracking (the explicit Track 2 gap) and gives the language real range
(map / filter / reduce).

- [x] `(vec-push v x)` on a growable `(vec t)` (`src/ast.rs` new `Expr::VecPush`,
      `src/json.rs`, `src/validate.rs`, `src/interp.rs`); `(idx v i)` / `(len v)`
      already existed and read a grown vec. Interp `Value::Vec` grows by append.
- [x] Affine move-tracking **DONE (validator)**: `vec-push` consumes a bare
      `(ref v)` target (marks the binding moved); a later plain use is
      `USE_AFTER_MOVE`, carrying the existing `insert_copy` repair which the loop
      now actually applies. `copy` revives (permitted on a moved binding, since v0
      `Value::Vec` is structurally copyable) — this is what makes the repair
      converge. `set` reassignment revives the binding, so the `(set acc (vec-push
      acc x))` accumulator pattern type-checks. `get`/`len`/`copy` borrow (do not
      consume). First place affine analysis does real work (the `moved` flag was
      previously never set). New tests: `vec_push_builds_vec_in_a_loop`,
      `vec_push_move_tracking_rejects_use_after_move`,
      `repair_loop_inserts_copy_for_use_after_move`. Suite 52→55.
- [~] Loop-carried aggregates: `set` + `vec-push` across a `loop` type-checks and
      runs in the **interpreter** (test `vec_push_builds_vec_in_a_loop`). Native
      lowering is **B2** — `vec-push` currently records an unsupported-lowering
      error (`self.err`), so `aury ll` skips the fn and native runs fail cleanly
      rather than diverging from the interpreter (parity invariant preserved).
- [ ] **B2 — native/wasm lowering + parity**: growable-vec runtime (dynamic
      alloc + capacity in `runtime/aury_rt.c`), descriptors, `src/lower.rs`
      `lower_vec_push`, then build/map/filter/reduce fixtures added to the native
      parity matrix (`tests/native_parity.aury`) across interp / native / wasm.
- [ ] Property generator + shrinker for growable vecs (shrink toward `[]`,
      element-wise; `src/spec.rs`).
- [ ] `map`/`filter`/`reduce` over a `(vec f64)` example + eval corpus case;
      skill docs (`AURY-FOR-AGENTS.md` node table + `vec-push`/affine section).
- **Acceptance (partial):** a use-after-move is mechanically repaired by the loop
      ✅; the `map`/`filter`/`reduce`-over-`(vec f64)` byte-identical across
      interp/native/wasm bar is **B2**.
- **Note:** shared / aliased mutable vecs stay out of scope — single-owner affine
      only (Track C covers aliasing).

## Track C — Real region semantics (stretch; retires the biggest caveat)

The largest standing "claimed but not implemented" caveat (README §Region
semantics): `region` / `copy` are immutable pass-throughs and native
allocations live for the process lifetime. This track makes regions real.

- [ ] Arena lifetime: `(region r …)` allocates into `r` and bulk-frees at scope
      exit; references carry their region explicitly (`src/interp.rs`,
      `src/lower.rs`, `runtime/aury_rt.c` arena allocator).
- [ ] Local borrow-graph alias pass (the proposal's decidable, region-scoped
      analysis): two overlapping `mut` references in one region are rejected
      pointwise (`ALIAS_CONFLICT`), with `split-region` / `insert-copy` mechanical
      repairs the loop applies (`src/validate.rs`, `src/repair.rs`).
- [ ] Escape checking: a reference must not outlive its region (`REGION_ESCAPE`),
      with a `widen-region` / `copy-out` repair.
- [ ] Differential parity + leak checks: region frees are observable (allocation
      count returns to baseline); fixtures across interp / native
      (`tests/native_parity.aury`).
- **Acceptance:** an aliasing violation is rejected pointwise and mechanically
      repaired; a program using explicit regions frees its arena at scope exit
      (verified by allocation accounting), matching across interp and native.
- **Alternative if C is descoped — C2 (SMT contracts):** discharge the
      linear-arithmetic contract fragment via an SMT backend, with
      `strengthen-precondition` / `weaken-postcondition` repairs, upgrading
      contracts from runtime assertions to proof for the decidable fragment
      (`src/spec.rs`, `src/validate.rs`). Heavier and less blocking than C1.

## Track D — Scaled evaluation + cross-language baseline

Built on A–C. The README names this the "next meaningful milestone." v0.1's
corpus is small and curated (most pass first-shot); v0.2 answers RQ1–RQ4 with
numbers.

- [ ] Grow `eval/corpus.json` to a larger, **uncurated** task set (target ≥ 30
      intents spanning effects, vecs, regions, floats, contracts) so first-shot
      pass rate is a real distribution, not a hand-tuned 7/8 (`src/eval.rs`,
      `eval/corpus.json`).
- [ ] Report a genuine **first-shot success rate** and a **mechanical-repair
      convergence rate** broken down by gate (parse / type / effect / region /
      intent) — Tracks A–C add the effect and region convergence cases the v0.1
      corpus lacked.
- [ ] **Cross-language baseline:** for each intent, generate the equivalent Python
      (and/or Rust), run it against the same oracle checks, and report first-shot
      correctness Aury-vs-baseline on a matched task set (`src/eval.rs`, new
      `eval/baseline/`). This is the comparison v0.1 explicitly deferred rather
      than fabricated.
- [ ] Deterministic, committed `eval/report.{md,csv}` regenerated by
      `aury eval`; `cargo test` gate asserts convergence rates and the presence of
      ≥1 mechanical repair per structural gate.
- **Acceptance:** `aury eval eval/corpus.json` prints per-gate first-shot and
      repair-convergence rates plus a cross-language first-shot comparison, and the
      test suite gates on them.

## Sequencing rationale

1. **Effects first (A)** — smallest diff, directly converts the eval's most
   embarrassing note ("effect-leak repair is NOT mechanically applied") into a
   demonstrated convergence. Every capability is also a new repair opportunity.
2. **Vecs second (B)** — gives the affine machinery its first real work and gives
   the language the range (map/filter/reduce) that makes a larger corpus
   meaningful.
3. **Regions third (C)** — highest surface area (interp + LLVM + runtime + alias
   analysis) and retires the biggest standing caveat; rides on the affine work in
   B. Descope to C2 (SMT contracts) if arena semantics prove too large.
4. **Evaluation last (D)** — only meaningful once A–C give the language range and
   the structural gates actually converge; then the cross-language baseline and
   per-gate convergence rates answer the research questions with numbers.
