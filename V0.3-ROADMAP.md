# Aury v0.3 roadmap

Tracking doc for the v0.3 language milestone. v0.2 closed the loop it set out to:
the repair loop now *mechanically converges* on the structural gates (effects,
regions), not just parse, and the language gained real range (growable vecs,
floats, arena regions). v0.2 left two honest gaps — a capability system that is
**checked but inert** (only `rng.*` actually executes; there is no `with` scope
and capabilities are ambient, not tokens), and the **generation baseline** the
project exists to answer, deferred because a hermetic harness cannot call a model
live. v0.3 has one theme:

> **Finish the capability system — make capabilities first-class, scoped, and
> executable with backend parity — then answer RQ1 for real by recording
> model-generated programs and replaying them deterministically.**

Ordered by dependency and leverage, anchored on capabilities (the highest-
confidence, most on-thesis work; the baseline is the marquee stretch). The
invariant for every track is unchanged: **the interpreter is the semantics, and
the native/wasm backend must produce the identical observable value**
(differential parity is a test failure, not a warning). Every new gate rejection
must ship with an *admissible mechanical repair* and a test proving the loop
applies it.

Legend: `[ ]` todo · `[~]` in progress · `[x]` done

## Track A — First-class capabilities + `with` scope (anchor; smallest on-thesis diff) — 🚧 A1 DONE

**A1 (landed):** the `with` form is real end to end (parse + typed-object/array
JSON + type + interp + native/wasm lowering + effect discharge), capabilities now
have two modes (ambient `rng` vs lexically-scoped `log`/…), and the closed loop
mechanically converges the new `CAPABILITY_NOT_IN_SCOPE` gate via
`wrap_in_capability_scope` — a convergence distinct from widening a row, because a
scoped capability is not row-grantable. Corpus case `log-scope`; parity fixture
`with-rng`; suite 61→64 (+3 Track A). Remaining for Track A: capability *values*
threaded as parameters (beyond lexical scope), and `wrap-in-capability-scope` as
an alternative repair on the ambient effect-leak menu.

The one place the language surface is still visibly stubbed. v0.2 fixed the
capability *vocabulary* (`KNOWN_CAPABILITIES` in `src/validate.rs`) and made the
three effect-row directions converge (widen / drop-unused / unknown-cap), but a
capability is **ambient** — `rng` is granted by declaring it in the `(effects …)`
row, with no lexical scope and no capability value. This track makes a capability
a *token* threaded through scope, which is the precondition for a sound `with`
form and for the `wrap-in-capability-scope` repair Track A deferred in v0.2.

- [~] Capability scope: a capability is granted lexically, not only ambiently.
      **Done** as a `with`-introduced scope (`caps_in_scope` in `src/validate.rs`)
      with two modes — ambient `rng` (row-granted) vs scoped `log`/`clock`/… (only
      `with` grants them, `is_scoped_capability`). **Remaining:** capability
      *values* threaded as parameters (the `cap_*` token path) beyond lexical
      scope.
- [x] `with` form: `(with (cap …) body)` grants capabilities to a lexical scope;
      parse + typed-object/array JSON + type + interp + lower (`src/ast.rs`,
      `src/json.rs`, `src/validate.rs`, `src/interp.rs`, `src/lower.rs`). The
      interpreter models the grant deterministically; lowering is transparent
      (capabilities have no runtime rep in v0), so interp/native/wasm agree. A
      `with` also **discharges** its caps from the function's row (a `with (rng)`
      draw is pure to callers) — parity fixture `with-rng`.
- [x] **`wrap-in-capability-scope` mechanical repair** (`src/validate.rs`
      `reject_capability_not_in_scope`; driver reuses the generic `?`-placeholder
      path in `src/loop_driver.rs`): a scoped-capability op used without an
      in-scope grant is a `CAPABILITY_NOT_IN_SCOPE` rejection targeting the op
      node with a `(with (cap) ?)` wrapper. The loop *applies* it and re-enters —
      a **new gate-convergence case** distinct from widen (widen is inadmissible
      for a scoped capability). The template the v0.1 Track 4 note and the v0.2
      Track A deferral both called out.
- [x] Tests (`tests/integration.rs`): `with_scope_discharges_ambient_effect`,
      `scoped_capability_op_requires_a_with_scope`,
      `repair_loop_wraps_in_capability_scope`. Existing widen/least-privilege
      tests unchanged (ambient `rng` path untouched).
- [x] Eval corpus: `log-scope` (`examples/agent/log-scope.aury`) — first-shot
      `effect✗` → wrapped in `(with (log) …)` → accepted, oracle 2/2. The effect
      gate now shows two converged cases (widen + wrap); corpus 12→13.
- **Acceptance:** a program that uses an effectful op without an in-scope grant
  is *mechanically repaired by the loop* (not hand-fixed) into a `with` scope,
  accepted end to end, and the eval corpus records the convergence.
- **Out of scope:** real OS execution / audited extern shims (still deferred);
  capabilities remain checked and deterministically modeled, not sandboxed.

## Track B — Effectful ops with backend parity (rides on A) — ✅ CORE DONE

Gives the scoped capabilities *observable behavior* under the same discipline
Track B (v0.2) applied to growable vecs: interp = native = wasm, byte-identical.
Two non-`rng` capabilities now execute; that meets the acceptance bar.

- [x] `clock.now` and `log.i64` as real effectful ops: interp (`src/interp.rs`) +
      native + wasm (`runtime/aury_rt.c` `aury_clock_now`/`aury_log_i64`,
      `src/lower.rs`), each gated by its scoped capability from Track A.
      `clock.now` is a **deterministic monotonic tick** (0, 1, 2, … — a
      per-run counter, not wall-clock) so parity is well-defined and the eval
      harness stays hermetic; `log.i64` emits to stderr (a side channel) and
      returns its argument, so the observable result stays parity-checkable.
- [ ] A deterministic `state` cell (`state.get` / `state.set`) — deferred; the
      two ops above already prove the machinery end to end. `state` (a *mutable*
      capability) is the natural next add if the token-threading needs exercising
      beyond read-only/echo sources.
- [x] Native/wasm parity fixtures (`tests/native_parity.aury`: `log-echo`,
      `clock-first`, `clock-ticks`) plus examples composing an op inside a `with`
      (`examples/agent/log-scope.aury`, `clock-stamp.aury`). Corpus gained
      `clock-scope`; the effect gate now shows **three** converged cases (widen +
      two wraps). Suite 64→67.
- **Acceptance MET:** two non-`rng` capabilities (`log`, `clock`) execute with an
  observable value byte-identical across interp / native (/ wasm where a wasi
  sysroot is present), each gated by an in-scope grant.
- **Note:** the point was to prove the capability *machinery* end to end, not to
  ship a broad stdlib — two well-tested ops beat six stubs.

## Track C — Region soundness completion (independent cleanup) — ✅ DONE

Retires the last standing region caveat (README §Region semantics). v0.2's arena
only freed regions the lowerer *proves* escape-free; the outer-`set` escape fell
back to process-lifetime. Both follow-ups landed.

- [x] Static **`REGION_ESCAPE`** rejection for the outer-`set`-escape case (a
      `set` of a binding declared *outside* the region), with a `copy_out`
      mechanical repair (`src/validate.rs` `check_region_escapes` /
      `reject_region_escape`; driver reuses the generic `?`-placeholder path).
      **Non-regressing:** the gate fires only for an outer-`set` whose value is
      not already a `copy`, and the repair (wrap in `copy`) never changes the
      observable value — so no existing program regressed (suite stayed green).
      Another Region-gate convergence case (the region gate now shows three:
      alias-split, use-after-move, copy-out).
- [x] Real deep-copy `copy` (`runtime/aury_rt.c` `aury_copy_value` walks the
      value descriptor; `src/lower.rs` lowers `copy` of an aggregate to it;
      scalars pass through; the interpreter is already independent via Rust
      ownership). A copied aggregate is a fresh, independent value.
- [x] The `copy-out` is made **freeable**: inside an arena region a `copy` is
      relocated to the **parent frame** (`aury_copy_to_parent`), so it survives
      the region's bulk free; `region_escapes` then treats an outer-`set` of a
      `copy` as non-escaping, so the region becomes arena-managed. A leak turned
      into a freed region.
- [x] Differential parity + leak checks: `tests/arena_accounting.c` gains a
      copy-out case (scratch freed, copied value survives); `tests/native_parity.aury`
      gains `region-escape` and `copy-vec`, byte-identical across interp/native.
      Corpus case `region-escape`; suite 67→70.
- **Acceptance MET:** an outer-`set`-escaping region is rejected and mechanically
  repaired so its arena is freed (allocation-accounting verified); `copy`
  produces an independent value, byte-identical across interp / native (/ wasm).
- **Note:** a `break` out of a region and a nested region still fall back to
  process-lifetime (sound, not freed) — the same conservatism, now the only
  remaining non-arena cases.

## Track D — Real generation baseline via record/replay (headline) — 🚧 HARNESS DONE (Phases 1–2)

The RQ1 the project exists to answer, deferred through v0.1 and v0.2 for an
honest reason: a deterministic, hermetic harness cannot invoke a model live. The
resolution is **record/replay** — decouple the (non-deterministic) generation
step from the (deterministic) scoring step. Baseline chosen: **Python**,
**first-shot only** (user decision) — the hardest, most conservative comparison.

- [x] **Record harness (offline, non-hermetic):** `eval/record/generate.py` +
      `eval/record/prompts/{aury,python}.md` + `eval/record/tasks.json` — a
      matched **language-neutral algorithmic** task set (add/gcd/factorial/fib/
      is_prime/isqrt; *not* the Aury-gate demos). The driver calls the Anthropic
      API for `k` generations per task per language and writes raw outputs
      (failures included) plus a provenance manifest (model, date, k, prompt
      hashes) under `eval/generated/<model>-<date>/`. **User runs this** with
      their API key.
- [x] **Replay (deterministic, in-harness):** `src/generated.rs` +
      `aury eval-generated <dir>` scores each recorded Aury generation through the
      repair loop (first-shot valid / accepted-after-repair) then the oracle, and
      each Python generation first-shot in a hermetic `python3` subprocess on the
      same oracle inputs. The genuine *generation-reliability* comparison v0.2's
      cross-*implementation* agreement explicitly was not.
- [x] Report + gate: `eval/generated-report.md`-style table (Aury vs Python:
      first-shot / converged / **final oracle-correct**) with a threats-to-validity
      block (sample size, model/date, familiarity-bias caveat). `cargo test`
      (`generated_replay_scores_are_deterministic`) gates on the replay being
      **deterministic over the committed synthetic sample**, not on a target
      win-rate. Python side skips hermetically when `python3` is absent.
- [x] Pipeline proven end to end: committed synthetic sample under
      `eval/generated/sample-2026-07-18/` (first-shot-valid, parse-repair
      convergence, and a valid-but-oracle-wrong case), replayed green.
- [ ] **The real recorded run (Phase 3):** run `generate.py` against a named
      model on a matched task set, commit the dated fixtures, regenerate the
      report. The one remaining piece — the generation step is non-hermetic, so it
      is run and committed deliberately.
- [ ] Corpus scale: grow `eval/record/tasks.json` toward ≥30 language-neutral
      tasks as the recording surface.
- **Acceptance (mostly met):** `aury eval-generated` reports a reproducible,
  honestly-labelled generation-reliability comparison (Aury vs Python) over
  recorded model outputs on a matched task set; the harness stays hermetic and
  deterministic because generation is recorded, not live. Remaining: the real
  recorded run at sample size.
- **Honesty guardrail (enforced in the report):** the number is small-sample and
  model/date-stamped, reported as evidence with its provenance, never as a
  headline win-rate divorced from sample size — and Python's familiarity
  advantage is stated inline, making this a conservative test for Aury.

## Sequencing rationale

1. **Capabilities first (A)** — smallest on-thesis diff, closes the one visibly
   stubbed surface, and adds a new mechanical-convergence case
   (`wrap-in-capability-scope`). Every capability is also a new repair
   opportunity, exactly as in v0.2 Track A.
2. **Effectful ops second (B)** — rides directly on A's token threading and gives
   the capability system observable behavior with backend parity; meaningless
   before A grants a scope to gate on.
3. **Regions third (C)** — independent of A/B, retires the last region caveat, and
   is a good parallel/filler track that adds one more convergence case.
4. **Baseline last (D)** — only meaningful once A–C give the language enough range
   for a matched task set to be fair, and it is the riskiest (needs an offline
   recording step). Record/replay is what makes it buildable at all without
   breaking the harness's determinism.
