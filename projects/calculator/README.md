# Aury Calculator → wasm → Vite

A full calculator written in **Aury**, compiled to a **wasm32-wasi reactor**
module, and driven by a **Vite** frontend. Every arithmetic operation the UI
performs is executed inside the wasm module — the JavaScript only marshals
values and renders.

```
calculator.json ──▶ (aury loop) ──▶ calculator.repaired.aury
   (authored JSON AST)                     │
                                           ▼  aury wasm-lib --export …
                             web/public/calculator.wasm
                                           │  fetch + instantiate
                                           ▼
                                 web/  (Vite frontend)
```

## What's in the Aury program

`calculator.json` is the authoring surface — the skill's typed-object JSON AST.
It is generated by **`gen.py`** (a Python builder), because the deeply-nested
mutable-loop / `if` / `let` trees are easier to keep balanced with a builder
than by hand-editing JSON. `aury loop` auto-applies any repairs to the ingested
canonical form; the program is accepted in **0 patches** and the native backend
is asserted to agree with the interpreter.

The calculator exercises both of Aury's v0.1 tracks:

- **Mutable loops (Track 2).** Every iterative integer function — `gcd`,
  `power`, `factorial`, `fibonacci`, `isqrt`, `is_prime` — is written with
  `let` + `set` + `loop` / `break` accumulators, no recursion and no internal
  `*_iter` helpers. The block-first shape (update the accumulator, *then* test
  for convergence with `if cond then (break x) else (block … unit)`) is what
  makes the convergence check fire on a *computed* value rather than the
  initial one.
- **f64 floats (Track 3).** A full parallel float function set is layered on top
  of the integer set, plus `cast`-based `i64 ↔ f64` conversions. Float division
  never traps: `fdivide(1, 0) = ∞`, `fdivide(0, 0) = NaN`. `is_nan` is just
  `f64.neq(a, a)` (since `NaN ≠ NaN`), and `fsqrt` returns `NaN` for negative
  input (obtained as `f64.div 0.0 0.0`, since there are no `inf`/`NaN`
  literals).

| Kind | Functions |
|------|-----------|
| Int binary `(i64,i64)->i64` | `add` `subtract` `multiply` `divide` `modulo` `percent` `average` `maximum` `minimum` `gcd` `lcm` `power` |
| Int unary `(i64)->i64` | `negate` `absolute` `square` `increment` `decrement` `double` `factorial` `fibonacci` `isqrt` |
| Int predicate `(i64)->bool` | `is_even` `is_prime` |
| Float binary `(f64,f64)->f64` | `fadd` `fsubtract` `fmultiply` `fdivide` `fmaximum` `fminimum` |
| Float binary `(f64,i64)->f64` | `fpower` (integer exponent; reciprocal for `exp < 0`) |
| Float unary `(f64)->f64` | `fnegate` `fabs` `fsquare` `freciprocal` `fsqrt` |
| Conversions / predicate | `to_float (i64)->f64` · `to_int (f64)->i64` · `is_nan (f64)->bool` |

`divide`/`modulo` are guarded against divide-by-zero (return `0`) so the module
never traps. The bounded integer functions (`factorial ≤ 20`, `fibonacci ≤ 92`,
`power`/`fpower` exponent `≤ 1024`) stay within `i64` range and keep loops
finite.

The `spec` block carries **12 properties**. The 8 integer ones (commutativity,
add/sub inverse, negate involution, gcd divides both, …) are unconditional.
The 4 float ones (fadd/fmultiply commutativity, fnegate involution, fabs
non-negative) are **NaN-guarded**: because `f64.eq(NaN, NaN)` is false, an
unguarded float property would be falsified on a NaN input, so each is wrapped
as `if (is_nan <result>) then true else <check>` — the property asserts the
check only when the value is finite.

## The wasm boundary

The module exports **38** `aury__<fn>` functions (plus `_initialize`) and links
with **zero imports**: the browser needs no WASI shim and instantiates it from
a plain `ArrayBuffer`. Across the boundary:

- `i64` and `bool` cross as wasm `i64` → JS **`BigInt`** (predicate returns
  come back as `0n`/`1n` and are wrapped to a JS boolean).
- `f64` crosses as a real wasm `f64` → JS **`Number`** — a scalar, not a
  linear-memory pointer, despite the `wasm-lib` "non-scalar" note (the note is
  conservative; `f64` is a wasm scalar and needs no marshaling).

`web/src/calculator.js` holds a per-export `SIGNATURES` table and marshals each
argument/return by type, so a single `bind()` path serves the whole mixed
integer/float surface.

## Edit and rebuild

`gen.py` is the source of truth — it builds `calculator.json`. To regenerate,
validate, property-test, and recompile the wasm:

```bash
python3 gen.py                  # gen.py -> calculator.json
./build-wasm.sh                 # calculator.json -> validate/property-test -> web/public/calculator.wasm
```

`build-wasm.sh` is a thin wrapper: it declares the export list and output path,
then calls the aury skill's `dev.sh` (author/repair loop) and `wasm-lib.sh`
(reactor builder). All the reusable work — ingest, the repair loop, and
`wasm32-wasi` toolchain detection — lives in the skill. To iterate on the AST
alone, run the loop directly:

```bash
./.claude/skills/aury/dev.sh projects/calculator/calculator.json factorial 10
```

The wasm build needs a `wasm32-wasi` toolchain (clang w/ the wasm target, a
wasi-libc sysroot, `wasm-ld`); the skill's bundled `wasm-toolchain.sh` locates
it (wasi-sdk or Homebrew llvm+lld+wasi-libc).

## Run the frontend

```bash
cd web
npm install
npm run dev        # http://localhost:5173  (or `npm run build` for a static dist/)
```

Vite serves `web/public/calculator.wasm` at `/calculator.wasm`. The page loads
it, calls `_initialize`, and every keypad/function button invokes an
`aury__*` export.

The UI has an **INT / FLT mode toggle** (the `INT`/`FLT` chip in the display).
INT mode is the integer calculator: BigInt entry, grouped digits, integer
binary/unary/predicate chips, and a `mod` key. FLT mode is the float
calculator: decimal entry, 12-significant-digit formatting (with `NaN`/`∞`
rendered literally), float binary/unary chips, and a `.` key (no float modulo).
The **→ float** / **→ int** convert chips call `to_float` / `to_int` and flip
the mode, carrying the value across the `i64 ↔ f64` cast boundary.
