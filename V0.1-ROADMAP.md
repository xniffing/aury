# Aury v0.1 roadmap

Tracking doc for the v0.1 language milestone. Ordered by dependency and
leverage. Each track lists the files it touches (verified against the current
tree) and its acceptance bar. The invariant for every track: **the interpreter
is the semantics, and the native/wasm backend must produce the identical
observable value** (differential parity is a test failure, not a warning).

Legend: `[ ]` todo · `[~]` in progress · `[x]` done

---

## Track 1 — Executable contracts (anchor; smallest diff, most on-thesis)

`requires`/`ensures` already parse into `Contract { pre, post }`
(`src/ast.rs:70`, `:396`) but are never executed. Wire them into a real gate.

- [ ] Stop collapsing multiple contracts into one pre-only node
      (`src/ast.rs:396-405` currently accumulates into a single contract).
- [ ] Execute `pre` before the call and `post` after in the interpreter, with
      `result` bound in `post` scope (`src/interp.rs`).
- [ ] Emit a structured `Rejection` on contract violation that re-enters the
      repair loop, with a new rejection kind + ranked repairs (`src/repair.rs`,
      `src/validate.rs`, `src/loop_driver.rs`).
- [ ] Validate contract expressions are `bool`-typed and reference only
      in-scope params + `result` (`src/validate.rs`).
- [ ] Vacuity: a `post` that ignores `result` should be flagged, mirroring the
      existing property vacuity check (`src/spec.rs`).
- [ ] Tests: violated pre, violated post, `result` binding, contract repair
      closure (`tests/integration.rs`).
- **Acceptance:** a function whose body violates its `ensures` is rejected with
      a repair; a corrected body is accepted; interp behavior is covered by tests.
- **Explicitly out of scope for v0.1:** SMT discharge / implication checking.
      Runtime assertion only.

## Track 2 — Mutable loops / accumulators

`Expr::Loop` exists (`src/ast.rs:123`) but there is no `set`/assignment, so all
iteration is recursion (the calculator's whole design). Add bounded mutable
iteration and let it exercise the currently-no-op affine/region machinery.

- [ ] Decide the form: a mutable `let` + `set` inside `loop`, or a `fold`
      builtin. (Leaning `set` + `loop` + `break value` — more general, and it
      stresses affine move-tracking, which is the point.)
- [ ] AST node(s) + JSON authoring surface + s-expr round trip
      (`src/ast.rs`, `src/json.rs`, `src/sexpr.rs`).
- [ ] Type/effect/affine validation: mutation of an affine binding, loop-carried
      state, `break`/loop-tail type agreement (`src/validate.rs`).
- [ ] Interpreter evaluation (`src/interp.rs`).
- [ ] LLVM lowering: basic-block loop with phi nodes for carried state
      (`src/lower.rs`) + differential parity fixture (`tests/native_parity.aury`).
- [ ] Port one calculator function from recursion to a loop as a demonstration.
- **Acceptance:** an iterative sum/factorial written with a mutable accumulator
      runs identically in interp, native, and wasm.

## Track 3 — f64 floats

Biggest type-system hole (`f64`/`f32` appear nowhere). Most cross-cutting work.

- [ ] `Type::F64` + parse/display (`src/types.rs`, `src/ast.rs`).
- [ ] Float literals in reader + JSON surface (`src/sexpr.rs`, `src/json.rs`).
- [ ] Builtins: `f64.add/sub/mul/div/neg/abs`, comparisons, `f64.to_str`,
      `i64<->f64` casts (`src/validate.rs`, `src/interp.rs`).
- [ ] **Define edge behavior before lowering** (matching the README's i64
      discipline): NaN/inf semantics, comparison of NaN, div-by-zero → inf vs
      trap. Document in README.
- [ ] LLVM lowering to `double` + runtime printing with a deterministic,
      interp-matching format (`src/lower.rs`, `runtime/aury_rt.c`,
      `src/value_io.rs`).
- [ ] Property generator for `f64` (`src/spec.rs`) — decide whether float
      properties use exact or epsilon equality.
- [ ] Differential parity across the NaN/inf/format edges (`tests/integration.rs`).
- **Acceptance:** a float algorithm (e.g. mean of a `(vec f64)`) matches across
      backends including the defined edge cases.

## Track 4 — Evaluation corpus (the README's stated "next meaningful milestone")

Not a language feature; measures whether the thesis holds. Meaningful only once
Tracks 1–3 give programs enough range to be non-trivial.

- [ ] Assemble an intent corpus: N tasks with natural-language intent + a
      reference oracle. Reuse `examples/agent/` and `projects/calculator/` as seeds.
- [ ] Harness that runs the closed loop per task and records: accepted/not,
      patches applied, repair-loop iterations, property/contract pass rate.
- [ ] Baseline comparison design (first-shot vs post-repair; Aury vs a mainstream
      target). Even a small honest table beats none.
- [ ] Report generation (markdown/CSV) so results are reproducible.
- **Acceptance:** `cargo run` (or a script) produces a repair-convergence table
      over the corpus that can go in the README's Evaluation section.

---

## Sequencing rationale

1. **Contracts first** — cheapest, reuses dormant AST, and every contract kind is
   also a new repair opportunity (the core thesis).
2. **Loops second** — removes the recursion-only distortion and finally exercises
   the affine/region design that is currently a pass-through no-op.
3. **f64 third** — highest surface area (interp + LLVM + runtime + parity), least
   novel relative to the thesis, so it rides on the infrastructure above.
4. **Evaluation last** — only meaningful once the language can express programs
   with real range; then it answers RQ1–RQ4 with numbers instead of assertions.
