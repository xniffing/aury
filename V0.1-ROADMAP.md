# Aury v0.1 roadmap

Tracking doc for the v0.1 language milestone. Ordered by dependency and
leverage. Each track lists the files it touches (verified against the current
tree) and its acceptance bar. The invariant for every track: **the interpreter
is the semantics, and the native/wasm backend must produce the identical
observable value** (differential parity is a test failure, not a warning).

Legend: `[ ]` todo · `[~]` in progress · `[x]` done

---

## Track 1 — Executable contracts (anchor; smallest diff, most on-thesis) — ✅ DONE

Design chosen: **function-level contracts** — `(requires E)` / `(ensures E)`
clauses inside the `fn` form (between the effect row and the body), rather than
the old unusable module-level `spec` `pre`/`post` (no function target, no
`result`). `ensures` binds the reserved name `result` to the return value.

- [x] Add `requires`/`ensures` to `FnDef`; parse the clauses in `build_fn`
      (`src/ast.rs`) and in the typed-object JSON surface (`src/json.rs`).
- [x] Enforce contracts in the interpreter: check `requires` on entry and
      `ensures` (with `result` bound) on exit; a violation traps. Both call
      paths go through `Interp::invoke_fn` (`src/interp.rs`).
- [x] Emit structured `Contract`-gate rejections that re-enter the repair loop:
      `POSTCONDITION_FALSIFIED` / `VACUOUS_CONTRACT` from the intent gate
      (`src/spec.rs`), `CONTRACT_NOT_BOOL` / `CONTRACT_IMPURE` from the type
      gate (`src/validate.rs`), wired into `loop_driver` + `cmd_test`.
- [x] Validate contract expressions are `bool`-typed and pure; scope (params,
      plus `result` for `ensures`) is enforced for free via `UNBOUND_REF`.
- [x] Vacuity: an `ensures` that never references `result` is flagged, mirroring
      the property vacuity check (`src/spec.rs`).
- [x] Active contract testing: `run_contract_tests` generates inputs, filters
      by `requires`, runs the fn, and shrinks in-domain counterexamples.
- [x] Tests: runtime enforcement, intent-gate falsification + shrinking,
      `requires` domain filtering, `result` scoping, bool/vacuity checks, JSON
      round trip (`tests/integration.rs`, 7 new tests; suite green at 32).
- **Acceptance met:** a function whose body violates its `ensures` is rejected
      with a repair and traps at runtime; a corrected body is accepted; behavior
      is covered end-to-end (`aury test`/`aury run`/`aury loop` verified).
- [x] Fixed the shared `shrink_one` so integer shrinking steps strictly toward
      zero (`n - n.signum()` instead of `n - 1`); counterexamples are now minimal
      magnitude (e.g. `x = -1`, not `-2`). Improves property shrinking too.
- **Out of scope (as planned):** SMT discharge / implication checking. Runtime
      assertion only.

## Track 2 — Mutable loops / accumulators — ✅ DONE

Design chosen: **`set` + `loop` + `break value`** (the roadmap's leaning
choice). A `let` binding is reassigned with `(set name value)`; `(break value)`
exits the nearest enclosing `loop` and makes the loop expression evaluate to
that value. A loop with no reachable `break` still diverges, exactly as before,
so existing recursion-only programs are unaffected.

- [x] Added `Expr::Break` and `Expr::Set` (`src/ast.rs`); parse `(break value)`
      / `(set name value)`, plus the typed JSON authoring surface (`src/json.rs`).
- [x] Interpreter: `Flow::Break` threads through the control-flow macro; `loop`
      catches `break` and yields its value; `set` mutates the binding in place
      (`src/interp.rs`). `match` arms now save/restore instead of cloning scope,
      so a `set` inside an arm mutates the enclosing binding (parity with native).
- [x] Validation with no inference: `loop` type is the agreed `break` value
      type (break-target stack); `break` outside a loop and disagreeing break
      types are rejected; `set` targets must be mutable locals of the right type
      (`src/validate.rs`). New gates: `SET_OF_PARAM`, `SET_UNBOUND`,
      `SET_TYPE_MISMATCH`, `BREAK_OUTSIDE_LOOP`, `BREAK_TYPE_MISMATCH`.
- [x] LLVM lowering: a loop with a `break` allocates a result slot and exit
      block; `break` stores + branches there; `set` stores to the binding's
      alloca (`src/lower.rs`). Loop-frame stack supports nesting.
- [x] Differential parity fixtures `loop-fact` / `loop-sum` / `loop-table`
      (nested) / `loop-empty` in `tests/native_parity.aury`, added to the native
      parity matrix; plus dedicated interp + gate tests (`tests/integration.rs`,
      9 new tests; suite green at 41).
- [x] Ported the calculator's `factorial` from a recursive `factorial_iter`
      accumulator to a mutable loop (`projects/calculator/calculator.json`);
      exported behavior is byte-identical, all 8 properties still pass, interp
      and native agree across `n ∈ {-1,0,1,2,5,10,20,21}`.
- **Acceptance met:** an iterative factorial with a mutable accumulator runs
      identically in interp and native across the full range including edges.
      (wasm uses the same target-independent lowering; a wasm-capable toolchain
      is required to rebuild `projects/calculator/web/public/calculator.wasm`.)
- **Note:** affine move-tracking over loop-carried aggregates remains a v0 no-op
      (there is no vector `push`); this track exercises the loop/break/set core
      on the i64 numeric surface.

## Track 3 — f64 floats

Biggest type-system hole (`f64`/`f32` appear nowhere). Most cross-cutting work.

- [ ] `Type::F64` + parse/display (`src/types.rs`, `src/ast.rs`).
- [ ] Float literals in reader + JSON surface (`src/sexpr.rs`, `src/json.rs`).
- [ ] Builtins: `f64.add/sub/mul/div/neg/abs`, comparisons, `f64.to_str`,
      `i64<->f64` casts (`src/validate.rs`, `src/interp.rs`).
- [ ] **Define edge behavior before lowering** (matching the README's i64
      discipline): NaN/inf semantics, comparison of NaN, div-by-zero → inf vs
      trap. Document in README.
- [ ] LLVM lowering to `double` + runtime printing with a deterministic,
      interp-matching format (`src/lower.rs`, `runtime/aury_rt.c`,
      `src/value_io.rs`).
- [ ] Property generator for `f64` (`src/spec.rs`) — decide whether float
      properties use exact or epsilon equality.
- [ ] Differential parity across the NaN/inf/format edges (`tests/integration.rs`).
- **Acceptance:** a float algorithm (e.g. mean of a `(vec f64)`) matches across
      backends including the defined edge cases.

## Track 4 — Evaluation corpus (the README's stated "next meaningful milestone")

Not a language feature; measures whether the thesis holds. Meaningful only once
Tracks 1–3 give programs enough range to be non-trivial.

- [ ] Assemble an intent corpus: N tasks with natural-language intent + a
      reference oracle. Reuse `examples/agent/` and `projects/calculator/` as seeds.
- [ ] Harness that runs the closed loop per task and records: accepted/not,
      patches applied, repair-loop iterations, property/contract pass rate.
- [ ] Baseline comparison design (first-shot vs post-repair; Aury vs a mainstream
      target). Even a small honest table beats none.
- [ ] Report generation (markdown/CSV) so results are reproducible.
- **Acceptance:** `cargo run` (or a script) produces a repair-convergence table
      over the corpus that can go in the README's Evaluation section.

## Sequencing rationale

1. **Contracts first** — cheapest, reuses dormant AST, and every contract kind is
   also a new repair opportunity (the core thesis).
2. **Loops second** — removes the recursion-only distortion and finally exercises
   the affine/region design that is currently a pass-through no-op.
3. **f64 third** — highest surface area (interp + LLVM + runtime + parity), least
   novel relative to the thesis, so it rides on the infrastructure above.
4. **Evaluation last** — only meaningful once the language can express programs
   with real range; then it answers RQ1–RQ4 with numbers instead of assertions.
